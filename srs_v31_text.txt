1. Introduction
1.1 Purpose
This Software Requirements Specification (SRS) defines the functional and non-functional requirements for Amira FinIQ, a Unified Financial Analytics Hub proposed for Mars, Incorporated. This document is intended for use by Amira Technologies' engineering and product teams, Mars's IT and digital leadership, and key business stakeholders who will evaluate, approve, and ultimately use the platform.
This SRS follows the IEEE 830-1998 and ISO/IEC/IEEE 29148:2018 standards for software requirements specifications.
This is the final unified specification incorporating all Databricks/FinSight data layer requirements and dual-mode operation (simulated and real data) into a single document.
1.2 Scope
Amira FinIQ is a unified financial analytics hub designed to consolidate and augment Mars, Incorporated's existing analytical capabilities, transitioning from a dispersed setup to a single, intelligent platform. The system will unify internal financial reporting and external competitive intelligence into one hub, enabling on-demand report generation, natural language queries, and enterprise-scale agent-based job processing.
The platform will integrate with Amira's existing Financial Forecasting and Marketing Analytics services, enabling forward-looking recommendations alongside historical analysis.
1.2.1 In Scope
Internal financial data ingestion via Databricks/FinSight (primary) and Excel workbooks (fallback)
External competitor document ingestion (earnings releases, presentations, transcripts)
Financial Modeling Prep (FMP) API integration for real-time competitor financial data, earnings transcripts, analyst estimates, M&A tracking, and ESG scores
Acquired research and third-party data integration
Natural language query interface for ad-hoc and structured reporting
Curated suggested prompt library with 18+ pre-configured analytical query templates (Appendix C)
Period End Summary generation (Summary, WWW, WNWW formats) with full configurability
Budget variance reporting (actual vs. replan from Databricks)
Peer-to-peer competitive benchmarking with quantitative comparison tables
Enterprise job board with agent queue, priority routing, and SLA management
Integration APIs for Amira Financial Forecasting and Marketing Analytics platforms
Dynamic, configurable user interface with real-time updates
Role-based access control (RBAC) aligned with Mars organizational hierarchy
Audit trail and data lineage for all generated reports
Caching and performance optimization for sub-second cached retrieval
Dual-mode operation: simulated data for development/testing, real Databricks data for production
1.2.2 Out of Scope
Direct SAP/ERP integration (financial data provided via Databricks/FinSight data layer; Excel exports retained as fallback during transition period)
Real-time streaming data (batch processing model)
Mobile native applications (responsive web application)
Multi-language report generation (English only initially)

2. Overall Description
2.1 Product Perspective
Mars currently operates two separate AI-powered analytical tools: a Period End Summary (PES) system for generating internal financial performance narratives, and a Competitive Intelligence (CI) system for analyzing competitor earnings documents. These systems operate independently, require engineering effort for even minor changes to report templates or data sources, and do not share a common data layer.
Amira FinIQ consolidates these capabilities into a unified platform while augmenting them with new functionality: on-demand reporting via natural language queries, an enterprise agent job board for scalable analytics processing, cross-platform intelligence through integration with Amira's Forecasting and Marketing Analytics services, a dynamic configurable user interface, and self-service configuration for business users.
2.1.1 Current System: Period End Summary (PES)
Function: Transforms raw financial Excel data into AI-generated executive performance summaries
Input: Preprocessed Excel workbooks (P&L, Product, Brand, NCFO sheets) from Azure Blob Storage
Processing: 6 parallel GPT-4.1 pipelines (one per KPI: Organic Growth, MAC Shape, A&CP Shape, CE Shape, Overhead Shape, NCFO)
Output: Markdown narratives with trend taglines, sub-unit rankings, and HTML KPI tables
Performance: 10-15 seconds first generation, < 1 second cached retrieval
Formats: Summary (balanced), What's Working Well (positive), What's Not Working Well (negative)
2.1.2 Current System: Competitive Intelligence (CI)
Function: Ingests competitor earnings documents and generates structured competitive analysis
Input: PDF documents (press releases, earnings presentations, prepared remarks)
Pipeline: Upload > Parse/Extract metadata > Chunk/Embed to Azure AI Search > Generate themed summaries > Peer-to-peer benchmarking
Output: Themed summaries (Organic Growth, Margins, Projections, Consumer, Products, etc.), P2P quantitative tables, Q&A chat
Infrastructure: Azure Blob Storage, Azure Document Intelligence, Azure OpenAI, Azure AI Search, Cosmos DB
2.1.3 Current System: FinSight Data Layer
Function: Mars's centralized Databricks-based financial data platform, providing the upstream data layer that feeds both PES and other analytical tools
Schema: finsight_core_model_mvp3 in the corporate_finance_analytics_dev catalog
Structure: 20 objects — 11 dimension tables, 5 fact tables, 3 precomputed analytics views, 1 system table
Fact tables: finiq_financial (39-col denormalized), finiq_financial_base (7-col normalized), finiq_financial_cons (9-col consolidated with multi-currency), finiq_financial_replan (18-col actual vs. budget), finiq_financial_replan_cons (6-col consolidated replan)
Precomputed views: P&L by entity, P&L by brand/product, NCFO by entity — each providing YTD and Periodic comparisons vs. Last Year
Storage: Delta Lake format on Azure Blob Storage (ABFSS protocol)
Significance: This data layer is the direct source of the Excel files currently used by PES. FinIQ will connect directly to these Databricks tables/views, bypassing the Excel export step.

3.2 Functional Requirements
3.2.1 FR1: Unified Data Ingestion Engine
The ingestion engine is the foundation of Amira FinIQ. It normalizes data from all sources into a unified data layer that the analytics engine queries against.
FR1.1: Internal Financial Data Ingestion (Databricks Primary)
The system shall ingest internal financial data via direct connectivity to Mars's Databricks/FinSight data layer (catalog: corporate_finance_analytics_dev, schema: finsight_core_model_mvp3). The primary ingestion path shall query the 3 precomputed views (finiq_vw_pl_entity, finiq_vw_pl_brand_product, finiq_vw_ncfo_entity) for Period End Summary analytics, and the normalized fact tables (finiq_financial_cons, finiq_financial_base) for ad-hoc queries. The 11 dimension tables shall be synchronized to the FinIQ data layer for entity hierarchy resolution, account structure, product taxonomy, and customer classification. Column mapping, KPI resolution, unit filtering (150+ organizational units via finiq_dim_entity), and derived metric calculations shall be applied automatically. Excel workbook ingestion (preprocessed_output_{Period}_{YearShort}.xlsx) shall be retained as a fallback path.
Priority: Critical | Category: Functional
FR1.2: Competitor Document Ingestion
The system shall ingest PDF documents (earnings releases, presentations, transcripts) via upload or automated scraping. Documents shall be parsed using Azure Document Intelligence, chunked, embedded, and indexed in a vector search store. Metadata extraction (company, quarter, document type) shall be automatic.
Priority: High | Category: Functional
FR1.3: Third-Party Data Connectors
The system shall provide a connector framework for integrating third-party data sources including commodity price feeds, acquired research databases, and market data providers. Each connector shall normalize data into the unified schema with source lineage metadata.
Priority: High | Category: Functional
FR1.4: Data Catalog & Lineage
All ingested data shall be cataloged with metadata (source, ingestion timestamp, schema version, quality score). Every derived metric and generated insight shall maintain a provenance chain back to its source data.
Priority: High | Category: Functional
FR1.5: Incremental & Scheduled Ingestion
The system shall support both manual uploads and scheduled ingestion jobs. Incremental ingestion shall detect changes and update only affected records. Ingestion status and errors shall be visible in the admin dashboard.
Priority: High | Category: Functional
FR1.6: Databricks Connection Management
The system shall maintain a configurable connection to the Databricks/FinSight environment with the following specifications: (a) Authentication via Azure service principal or managed identity with least-privilege read access, (b) Connection parameters (catalog, schema, warehouse/cluster endpoint) configurable via admin interface without code deployment, (c) Connection health monitoring with automatic retry (3 attempts, exponential backoff), (d) Query timeout of 60 seconds with graceful fallback to cached data, (e) Connection pooling with a maximum of 10 concurrent Databricks sessions, (f) All queries logged with execution time, row count, and data freshness timestamp for audit purposes.
Priority: High | Category: Functional
3.2.2 FR2: Financial Analytics & Reporting Engine
The analytics engine generates structured financial reports from the unified data layer. It evolves the current PES system into a more flexible, configurable architecture. When Databricks connectivity is available, the engine queries precomputed FinSight views directly, eliminating the Excel preprocessing step.
FR2.1: Period End Summary Generation (Databricks-Enabled)
The system shall generate executive performance summaries for any organizational unit, time period, and output format. When Databricks connectivity is available, the system shall query the precomputed FinSight views directly: finiq_vw_pl_entity for P&L by entity (maps to P&L sheet), finiq_vw_pl_brand_product for P&L by brand/product (maps to Product and Brand sheets), and finiq_vw_ncfo_entity for NCFO (maps to NCFO sheet). Each view provides Date_ID, Entity_Alias, Account_Alias, and four value columns (YTD_LY_Value, YTD_CY_Value, Periodic_LY_Value, Periodic_CY_Value) that directly feed the KPI narrative generation pipeline. This eliminates the Excel preprocessing step from the current 10-step pipeline. Six KPI narratives (Organic Growth, MAC Shape, A&CP Shape, CE Shape, Controllable Overhead Shape, NCFO) shall be generated in parallel, with trend analysis taglines injected from multi-period historical data. Output formats: Summary, What's Working Well, What's Not Working Well. When Databricks is unavailable, the system shall fall back to the Excel-based pipeline.
Priority: Critical | Category: Functional
FR2.2: Configurable KPI Framework
Business users shall be able to define, modify, and add KPIs without code changes. Each KPI definition shall include: formula, data sources, display name, narrative prompt template, and threshold parameters for trend analysis. KPI changes shall take effect on the next report generation. The finiq_account_formula table shall serve as the authoritative source for KPI calculation logic, enabling dynamic KPI configuration from Databricks metadata.
Priority: High | Category: Functional
FR2.3: Sub-Unit Ranking & Driver Analysis
For each KPI, the system shall automatically identify and rank sub-units (child business units) by performance for both Periodic and YTD timeframes. Driver analysis shall explain what is driving each sub-unit's performance, with explicit rank labels (RANK 1, TOP 3, BOTTOM 3).
Priority: High | Category: Functional
FR2.4: Interactive KPI Tables
Every generated summary shall include interactive HTML KPI tables with the underlying data, enabling users to cross-reference AI-generated narratives with source numbers. Tables shall support sorting, filtering, and drill-down.
Priority: High | Category: Functional
FR2.5: Custom Report Builder
Users shall be able to compose custom reports by selecting KPIs, organizational units, time periods, comparison bases (vs LY, vs Budget, vs Forecast), and output sections. Reports shall be saveable as templates for reuse.
Priority: High | Category: Functional
FR2.6: Export Capabilities
Reports shall be exportable to PDF, Word (.docx), PowerPoint (.pptx), Excel (.xlsx), and email. PowerPoint exports shall use Mars-branded templates with charts and tables pre-formatted for board presentation.
Priority: High | Category: Functional
FR2.7: Budget Variance Reporting
The system shall generate actual-vs-budget (replan) variance reports by querying the finiq_financial_replan table. For each organizational unit and time period, the system shall display: Actual_USD_Value, Replan_USD_Value, variance (Actual - Replan), and variance percentage. Both USD and Local currency values shall be supported. Variance reports shall be available as a standalone report type and as an enrichment layer on Period End Summaries (e.g., 'Petcare Organic Growth was 6.3% vs. a replan target of 5.5%, exceeding budget by 80 basis points'). The Submission_Type_ID column shall be used to distinguish between different forecast revisions.
Priority: High | Category: Functional
3.2.3 FR3: Competitive Intelligence Module
The CI module evolves the current standalone Competitive Intelligence tool into an integrated component of the analytics hub, with cross-referencing to internal data.
FR3.1: Themed Competitor Summaries
The system shall generate structured summaries of competitor earnings organized by themes: Organic Growth, Margins, Projections, Consumer Trends, Product Launches, and Miscellaneous. Each bullet shall include source references (clickable links to source document sections).
Priority: High | Category: Functional
FR3.2: Peer-to-Peer Benchmarking
The system shall generate quantitative P2P comparison tables across predefined competitor peer groups (e.g., Petcare segment: Mars, Nestle PetCare, Colgate-Palmolive, Freshpet, IDEXX, J.M. Smucker). Metrics shall include OG%, Price, Volume, Mix, Adj. Core Operating Profit %, with both Quarterly and YTD views. Narrative analysis shall identify leaders, laggards, and drivers.
Priority: High | Category: Functional
FR3.3: Internal-External Cross-Reference
The system shall enable queries that combine internal Mars data with competitor data. Example: 'Compare our Petcare organic growth vs Nestle PetCare and Freshpet for Q2 2024, and show our forecast for Q3.' This requires joining the unified data layer across internal and external sources.
Priority: Critical | Category: Functional
FR3.4: Competitor Monitoring & Alerts
The system shall support automated monitoring of competitor document sources. When new earnings documents are detected, the system shall automatically ingest, process, and notify relevant users. Users shall configure alert rules based on competitor, metric thresholds, or keywords.
Priority: High | Category: Functional
3.2.4 FR4: Natural Language Query Interface
The query interface is the primary user interaction layer. Users ask questions in natural language and receive structured, sourced answers.
FR4.1: Conversational Query Engine
The system shall accept natural language queries and return structured answers with supporting data tables and source references. Examples: 'What was Nestle's organic growth in Q2 2024?', 'Show me our Petcare MAC Shape vs last year by division', 'Which competitor had the highest margin improvement this quarter?'
Priority: High | Category: Functional
FR4.2: Multi-Turn Conversations
The system shall maintain conversation context across multiple turns, allowing follow-up questions ('Break that down by region', 'What about YTD?', 'Compare that with our forecast'). Conversation history shall be persisted per user session.
Priority: High | Category: Functional
FR4.3: Query Intent Classification
The system shall classify queries into types: factual lookup, comparison, trend analysis, recommendation, report generation, or job submission. Each type shall route to the appropriate processing pipeline (direct answer, analytics engine, forecasting API, or agent queue).
Priority: High | Category: Functional
FR4.4: Source Attribution
Every answer shall include source attribution: document name, section, page number (for PDFs), or cell reference (for Excel data). Users shall be able to click through to the source material.
Priority: High | Category: Functional
FR4.5: Suggested Prompt Library
The system shall provide a curated library of suggested prompts — pre-configured analytical query templates that users can select and execute with one click. Each suggested prompt shall include: a natural language template with dynamic variables ({unit}, {current_year}, {current_period}, {current_quarter}), a human-readable description, associated KPI tags, an organizational unit scope, and a usage counter (runs) for analytics. The prompt library shall be managed as a Cosmos DB collection (see Appendix C for the initial catalog of 18 prompts). Suggested prompts shall be displayed contextually in the query interface based on the user’s organizational unit, role, and recent query history. The system shall support the following variable resolution at execution time:
{unit} — resolved from the user’s active organizational unit selection or RBAC-scoped default entity (e.g., “MARS INC”, “GBU PETCARE”)
{current_year} — resolved to the current fiscal year from finiq_date
{current_period} — resolved to the most recent closed period from finiq_date (via Dimensions_View_Date_Map)
{current_quarter} — resolved to the current fiscal quarter number
Administrators shall be able to add, edit, disable, and re-order suggested prompts via the administration console. Prompts shall be taggable (e.g., #SuggestedbyFinIQTeam, #Popular, #NewThisPeriod) for filtering and organization. The system shall track prompt usage (runs counter) to surface the most popular prompts and inform the recommendation engine. Users shall be able to save custom prompts to their personal library and share them with their team.
Priority: High | Category: Functional
FR4.6: Prompt Variable Resolution Engine
The system shall implement a variable resolution engine that dynamically substitutes template variables in suggested prompts prior to execution. The engine shall: (a) resolve {unit} against finiq_dim_entity with fallback to the user’s default unit, (b) resolve {current_year} and {current_period} from the most recent Date_ID in the active FinSight views, (c) resolve {current_quarter} from the finiq_date dimension, (d) validate that all variables are resolved before submission to the query engine, and (e) allow users to override any resolved variable before execution. The engine shall support compound prompts that reference multiple dimensions (e.g., brand within unit within period). Template variable resolution shall complete in < 200ms.
Priority: High | Category: Functional
3.2.5 FR5: Enterprise Job Board & Agent Queue
The job board is a critical differentiator. It transforms Amira FinIQ from a single-user query tool into an enterprise-scale analytics service that can handle hundreds of concurrent requests.
FR5.1: Job Submission
Users shall submit analytical jobs via: (a) the query interface (complex queries auto-routed to queue), (b) a dedicated job submission form with fields for query, priority, deadline, recipients, and output format, (c) email integration (forward a question to finiq@mars.com), (d) API for programmatic submission.
Priority: Critical | Category: Functional
FR5.2: Agent Pool & Specialization
The system shall maintain a pool of specialized AI agents, each optimized for specific query types: Period End Summary Agent, Competitive Analysis Agent, Forecasting Agent, Marketing Analytics Agent, Ad-Hoc Query Agent, and Data Quality Agent. Agents shall be horizontally scalable based on queue depth.
Priority: High | Category: Functional
FR5.3: Priority Routing & SLA Management
Jobs shall be assigned priority levels (Critical, High, Medium, Low) with corresponding SLA targets. Critical jobs: < 2 minutes. High: < 10 minutes. Medium: < 30 minutes. Low: < 2 hours. The system shall monitor SLA compliance and escalate breaches to administrators.
Priority: High | Category: Functional
FR5.4: Job Lifecycle Management
Each job shall have a visible lifecycle: Submitted > Queued > Assigned (to agent) > Processing > Review > Completed/Failed. Users shall see real-time status updates. Failed jobs shall include error details and retry options.
Priority: High | Category: Functional
FR5.5: Job Dashboard
An enterprise dashboard shall show: active jobs by status, agent utilization, SLA compliance rates, average processing times by query type, top requestors, and historical trends. Filterable by department, priority, agent type, and date range.
Priority: High | Category: Functional
FR5.6: Scheduled & Recurring Jobs
Users shall schedule recurring jobs (e.g., 'Generate Petcare Period End Summary on the 5th of every month' or 'Run competitive benchmarking weekly for all peer groups'). The scheduler shall support cron-like expressions with timezone awareness.
Priority: High | Category: Functional
FR5.7: Collaborative Review & Approval
Jobs producing sensitive or high-visibility reports shall support a review workflow: the AI agent generates a draft, a designated reviewer approves/edits, and the final version is distributed. Review assignments shall be configurable per report type and organizational unit.
Priority: High | Category: Functional
3.2.6 FR6: Integration Layer
Amira FinIQ integrates with Amira's existing platform services to provide forward-looking recommendations alongside historical analysis.
FR6.1: Amira Financial Forecasting Integration (Three-Way Comparison)
The system shall call the Amira Financial Forecasting API to enrich reports with forward-looking projections. In addition, the system shall leverage the finiq_financial_replan table for Mars's internal replan/budget data. This enables a three-way comparison: Actual (from finiq_financial_cons) vs. Replan (from finiq_financial_replan) vs. Amira Forecast (from Forecasting API). Example output: 'Q2 Organic Growth was 6.3% (Actual) vs. 5.5% (Mars Replan) vs. 6.1% (Amira Forecast). Actual exceeded both targets. Based on current trends, Amira projects 5.8% for Q3.' Confidence intervals shall be displayed for the Amira Forecast; replan figures shall be labeled as Mars internal budget targets. Forecasts shall be clearly labeled as projections.
Priority: Critical | Category: Functional
FR6.2: Amira Marketing Analytics Integration
The system shall call the Amira Marketing Analytics API to correlate financial performance with marketing spend and effectiveness. Example: 'Royal Canin posted 11.3% OG in Q2. Marketing analytics shows a 15% increase in digital ad spend with a 2.3x ROAS, suggesting the growth is partially marketing-driven.' Marketing data shall be available as a dimension in custom reports.
Priority: Critical | Category: Functional
FR6.3: Unified Recommendation Engine
When both forecasting and marketing data are available, the system shall generate actionable recommendations: 'Based on forecasted cocoa price increases of 12% in Q3 and your current marketing ROI on Confectionery brands, consider shifting 5% of Confectionery marketing budget to Petcare, which shows stronger pricing power.' Recommendations shall be flagged as AI-generated suggestions.
Priority: High | Category: Functional
FR6.4: External API Gateway
The system shall expose a RESTful API gateway for third-party integrations. Endpoints shall include: report generation, job submission, query execution, data catalog browsing, and webhook registration for event notifications.
Priority: High | Category: Functional
FR6.5: Databricks Data Freshness Monitoring
The system shall monitor data freshness in the Databricks/FinSight schema by tracking: (a) the most recent Date_ID available in each fact table and view, (b) the Last_Change timestamp from finiq_rls_last_change for security policy currency, (c) row counts per entity and period for completeness validation. Data freshness indicators shall be displayed on all reports (e.g., 'Data as of Period 6, 2026'). Stale data alerts shall be triggered when expected period data is not available within a configurable window (default: 5 business days after period close).
Priority: High | Category: Functional
3.2.7 FR7: Administration & Configuration
FR7.1: Template Management
Administrators shall manage report templates through a visual editor. Templates define: sections, KPI selection, narrative tone, data filters, and output format. Templates shall be versionable with rollback capability.
Priority: High | Category: Functional
FR7.2: Organizational Hierarchy Management
The system shall maintain Mars's organizational hierarchy (Mars Inc > GBUs > Divisions > Regions > Sub-units) with the ability to add, modify, or reorganize units. Hierarchy changes shall automatically propagate to report generation and access control.
Priority: High | Category: Functional
FR7.3: Peer Group Configuration
Administrators shall define and modify competitor peer groups for benchmarking. Each peer group specifies: segment name, included companies, relevant metrics, and data source mappings.
Priority: High | Category: Functional
FR7.4: Prompt Management
LLM prompts shall be managed as configurable assets, not hardcoded. Each prompt shall have: system message, dynamic sections (output format selection), user message template, and injected data variables. Prompt changes shall take effect immediately without deployment.
Priority: High | Category: Functional
FR7.5: User & Access Management
RBAC aligned with Mars organizational hierarchy. Roles: Admin, Analyst, Viewer, API Consumer. Data access scoped by organizational unit (a Petcare analyst cannot access Snacking financials unless explicitly granted). SSO integration with Mars identity provider.
Priority: High | Category: Functional
FR7.6: Databricks Connection Administration
Administrators shall manage Databricks/FinSight connectivity through a dedicated admin interface. Configuration shall include: (a) Databricks workspace URL and warehouse/cluster endpoint, (b) Authentication credentials (service principal client ID/secret or managed identity), (c) Catalog and schema selection with object discovery (list available tables/views), (d) View-to-report mapping configuration (which views feed which report types), (e) Dimension table sync schedule (frequency for refreshing local copies of dimension data), (f) Connection test functionality with diagnostic output, (g) Fallback toggle: switch between Databricks primary mode and Excel fallback mode without code deployment.
Priority: High | Category: Functional
3.2.8 FR8: Dynamic User Interface
The user interface shall be dynamic, adaptive, and configurable — enabling users to personalize their experience, interact with data in real time, and work within a context-aware workspace that adjusts its layout and components based on user intent and workflow stage. The UI shall support progressive disclosure, multi-panel workspaces, and persistent state management.
FR8.1: Configurable Dashboard Layout
Users shall be able to arrange, resize, and pin dashboard widgets (KPI cards, charts, tables, job queue status panels) via drag-and-drop. Dashboard configurations shall be saved per user and shareable across teams. Administrators shall define default layouts per role.

7.4 Standard Competitive Intelligence Views
The CI module shall deliver the following standard analytical views, each designed around established strategic analysis frameworks. Views are prioritized based on Mars stakeholder feedback and aligned to the FinIQ user personas (Finance Analyst, Strategy Director, Marketing VP, CFO).
7.4.1 SWOT Analysis View — Detailed Specification
The SWOT view shall generate a structured Strengths-Weaknesses-Opportunities-Threats matrix for each competitor in the monitored universe. The SWOT is auto-generated quarterly using the following data pipeline:
Strengths/Weaknesses: Derived from FMP financial ratios (margins, growth, ROE), employee count trends, analyst consensus, and earnings transcript sentiment. GPT-4.1 synthesizes financial signals into natural language SWOT bullet points.
Opportunities/Threats: Derived from FMP news feed, press releases, M&A activity, SEC filings, and earnings call forward-looking language. Supplemented by curated industry intelligence from Databricks/FinSight.
Users can manually override or annotate any SWOT element. All changes are versioned and attributable. SWOT history is maintained for trend analysis (e.g., how a competitor’s profile has shifted over 8 quarters).
7.4.2 Porter’s Five Forces View
This view provides a quantified assessment of industry competitive dynamics using Porter’s Five Forces framework, segmented by Mars business unit (confectionery, pet care, food products):
Competitive Rivalry: Quantified via HHI (Herfindahl–Hirschman Index) calculated from FMP market cap data; number of competitors; revenue concentration; margin pressure trends.
Threat of New Entrants: Assessed via FMP fundraiser/IPO data, private equity investments in CPG, e-commerce DTC brand proliferation signals from news feeds.
Threat of Substitutes: Tracked through cross-category demand signals, consumer behavior data from FinSight, and plant-based/functional food trend analysis.
Supplier Power: Commodity price tracking (cocoa, wheat, meat proteins) via FMP commodity endpoints; supply chain concentration metrics from FinSight.
Buyer Power: Retail concentration analysis, private-label penetration trends, pricing elasticity models from Mars internal trade promotion data.
7.4.3 Earnings Call Intelligence View
This view applies natural language processing to earnings call transcripts sourced from the FMP Earnings Transcript API. For each competitor earnings call, the system shall produce:
Sentiment Analysis: Overall sentiment score (positive/neutral/negative) for management prepared remarks and Q&A section separately. Tracked over time to identify tone shifts.
Strategic Keyword Extraction: Frequency analysis of strategic terms (e.g., “premiumization,” “innovation pipeline,” “pricing power,” “cost savings,” “market share”). Compared across quarters and peers.
Guidance Language Analysis: Extraction of forward-looking statements, capex guidance, M&A hints, and segment-specific commentary. Flagged if language shifts significantly vs. prior quarter.
Competitive Mention Tracking: Detection of explicit or implicit references to Mars, Mars brands, or Mars-adjacent categories in competitor calls.
7.4.4 Financial Benchmarking Dashboard
An interactive dashboard providing side-by-side financial comparison across the competitor universe. Metrics include revenue, revenue growth (YoY/QoQ), gross margin, operating margin, net margin, ROE, debt-to-equity, free cash flow yield, and enterprise value. All sourced from FMP Financial Statements, Ratios, and Key Metrics endpoints. Users can customize the peer set, time horizon, and metric selection.
7.4.5 Competitive Positioning Map
An interactive 2D scatter plot positioning competitors along user-selectable dimensions (e.g., X-axis = revenue growth rate; Y-axis = operating margin). Bubble size represents market capitalization. Users can drill into any competitor bubble to see the full profile card. The map auto-updates quarterly with FMP data.
7.4.6 M&A and Strategic Moves Tracker
A chronological timeline of competitor M&A transactions, joint ventures, divestitures, and major strategic announcements. Data sourced from FMP M&A Search, Press Releases, and SEC Filings endpoints. Real-time alerts are pushed to subscribed users when a competitor in the monitored universe announces a material transaction.

Amira FinIQ shall support two operating modes — simulated data mode and real data mode — switchable via a single configuration toggle. This enables development and testing to proceed independently of Databricks access provisioning, and ensures the system can be validated end-to-end before connecting to production data.

8.2 Configuration Toggle
A single environment variable or admin setting (e.g., DATA_MODE=simulated|real) shall switch the entire system between modes
The toggle shall affect all data ingestion, query, and reporting paths simultaneously
Switching modes shall not require code changes or redeployment
The current mode shall be clearly indicated in the UI (e.g., a banner: 'Running on simulated data')

Generate synthetic data for all 20 Databricks objects matching exact column names, data types, and array structures

Product taxonomy: generate 100+ composite items across multiple brands, segments, and categories
Customer data: generate 50+ customers with hierarchical relationships and channel/format classifications
Date dimension: populate for FY2024 and FY2025 with Mars fiscal calendar periods
View outputs: simulate the precomputed view outputs (YTD_LY, YTD_CY, Periodic_LY, Periodic_CY) with mathematically consistent values

Simulated data shall be loadable into a local SQLite or PostgreSQL database for development and testing

Integration tests shall run against both simulated and real Databricks environments with the same configuration toggle
Simulated data shall be deterministic (seeded random) so test results are reproducible
8.5 Mode Parity
All features shall behave identically in both modes. The only differences shall be:
Data source connection (local DB vs. Databricks)
UI mode indicator (banner showing simulated vs. real)
Data freshness values (synthetic timestamps vs. real)
No feature shall be available in one mode but not the other. This ensures that validation in simulated mode provides confidence for real data operation.

C.7 Prompt Analytics
The system shall track the following analytics per suggested prompt:
Total executions (runs counter, incremented on each use)
Unique users per prompt per period
Average execution time and success rate
Most common variable overrides (e.g., which units are most queried)
Prompt abandonment rate (selected but not executed)
These analytics shall be available in the administration dashboard and used by the recommendation engine (FR4.5) to surface relevant prompts to users.
